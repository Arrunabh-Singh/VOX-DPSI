## Task 3 — Wire WhatsApp and SMS Notifications to Status Changes

### 3a. Current state analysis:
- `notifications.js` already calls WhatsApp functions (`notifyComplaintUpdate`, `notifyCouncilAssignment`, etc.) in all status change handlers.
- However, `whatsapp.js` line 27 has `if (true) return` which disables ALL WhatsApp sends.
- SMS via MSG91 is NOT wired for complaint status updates. `msg91.js` only exports `sendSmsOtp` (for VPC OTP). No function exists for complaint status SMS.

### 3b. Fix WhatsApp disable:
Replace hardcoded `if (true) return` in `whatsapp.js:27` with a conditional based on env:
```js
if (!process.env.TWILIO_ACCOUNT_SID || !process.env.TWILIO_AUTH_TOKEN) {
  return  // Twilio not configured — silently skip
}
```
This allows WhatsApp to work in production where env vars are set, while staying disabled in dev if not configured.

### 3c. Add SMS status notification support:
**In `server/services/msg91.js`:**
- Add new function `sendComplaintStatusSms(phone10digit, complaintNo, newStatus, handlerName)` that sends an SMS template via MSG91.
- Use the `MSG91_SMS_STATUS_TEMPLATE_ID` env var if available; otherwise log a warning that SMS would be sent.
- Structure similar to `sendWhatsAppStatusUpdate` but for SMS.

**In `server/services/notifications.js`:**
- Import `sendComplaintStatusSms` from `./msg91.js`
- In `notifyStatusChange` (line 47), after the WhatsApp call (line 70), add:
  ```js
  // SMS — student's own number (non-blocking)
  if (studentPhone) {
    sendComplaintStatusSms(studentPhone, complaintNo, newStatus, 'the assigned handler').catch(() => {})
  }
  ```
- Keep all calls wrapped in try/catch or `.catch()` to be non-blocking.

### 3d. Verify:

- `node --check server/services/msg91.js` → SYNTAX OK (after adding function)
