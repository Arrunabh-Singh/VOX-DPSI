## PHASE 2: BUG-01 — CREAM/BEIGE DEAD ZONE ON VIEWPORT (HIGH)

### Problem
`client/src/index.css` line 24 currently reads:
```css
html, body, #root { height: 100vh; background-color: #1a2e16; }
```
`height: 100vh` is a fixed ceiling — content overflows it, leaving ~390px cream band on the right and ~280px cream below the footer. The `background-color` on `body` (line 30: `var(--color-bg)` which is `#eae1c4` = cream) leaks through the uncovered viewport.

### Current state (line 24 of index.css)
```css
html, body, #root { height: 100vh; background-color: #1a2e16; }
```

### Fix
Replace line 24 ONLY. Change:
```css
html, body, #root { height: 100vh; background-color: #1a2e16; }
```
To:
```css
html { min-height: 100%; background-color: #1a2e16; }
body, #root { min-height: 100vh; width: 100%; }
```

**Why this works:**
- `min-height` expands to content height instead of cutting it off
- `width: 100%` ensures the `#root` div fills the full viewport horizontally
- `background-color: #1a2e16` on `html` covers the entire document behind all scrollable content
- The `body` background override (line 30: `var(--color-bg)`) only affects the body element itself, not the viewport behind it

## PHASE 3: BUG-05 — "COMPLAINT NOT FOUND" AFTER SUBMISSION (CRITICAL)

### Problem
After student submits a complaint (receives "VOX-XXXX" confirmation), clicking "Complaint Details" navigates to `/complaints/{uuid}` and shows either:
- "Complaint not found." with a toast error
- OR blank page silently

The complaint IS in the database (visible in dashboard list). Only the single-complaint detail endpoint fails.

### Root Cause Analysis
The `GET /api/complaints/:id` route at line 385 of `server/routes/complaints.js` has two potential failure points:

**Issue 1 — Student access check (line 405):**
```js
if (role === 'student' && complaint.student_id !== userId) {
  return res.status(403).json({ error: 'Access denied' })
}
```
When Supabase joins `student:student_id(id, name, ...)`, the `complaint.student_id` field may contain a joined object instead of the raw UUID string. Comparing `object !== string` would always be true, denying access even to the student who owns the complaint.

**Issue 2 — Council member access (line 417):**
```js
if (role === 'council_member' && complaint.assigned_council_member_id !== userId) {
```
If `assigned_council_member_id` is NULL (complaint not yet assigned), this check will fail for NULL !== userId. The delegation fallback doesn't handle NULL properly.

### Fix (Step 1 — Fix student access check at line 405)
Replace lines 404-407:
```js
// Access control
if (role === 'student' && complaint.student_id !== userId) {
  return res.status(403).json({ error: 'Access denied' })
}
```
With:
```js
// Access control — handle Supabase join which may replace FK with object
if (role === 'student') {
  const actualStudentId = (complaint.student_id && typeof complaint.student_id === 'object')
    ? (complaint.student_id.id || complaint.student?.id)
    : complaint.student_id
  if (actualStudentId !== userId) {
    return res.status(403).json({ error: 'Access denied' })
  }
}
```

### Fix (Step 2 — Fix council member access at line 417)
Replace lines 417-430:
```js
if (role === 'council_member' && complaint.assigned_council_member_id !== userId) {
  // #20 — allow delegate access
  const today = new Date().toISOString().slice(0, 10)
  const { data: delegation } = await supabase
    .from('delegation_rules')
    .select('id')
    .eq('delegate_id', userId)
    .eq('delegator_id', complaint.assigned_council_member_id)
    .lte('start_date', today)
    .gte('end_date', today)
    .limit(1)
    .maybeSingle()
  if (!delegation) return res.status(403).json({ error: 'Access denied' })
}
```
With:
```js
if (role === 'council_member') {
  const actualAssignedId = (complaint.assigned_council_member_id && typeof complaint.assigned_council_member_id === 'object')
    ? (complaint.assigned_council_member_id.id || complaint.council_member?.id)
    : complaint.assigned_council_member_id
  const isDirectlyAssigned = actualAssignedId && actualAssignedId === userId
  if (!isDirectlyAssigned) {
    // #20 — allow delegate access
    const today = new Date().toISOString().slice(0, 10)
    const { data: delegation } = await supabase
      .from('delegation_rules')
      .select('id')
      .eq('delegate_id', userId)
      .eq('delegator_id', actualAssignedId)
      .lte('start_date', today)
      .gte('end_date', today)
      .limit(1)
      .maybeSingle()
    if (!delegation) return res.status(403).json({ error: 'Access denied' })
  }
}
```

### Fix (Step 3 — Add null guard to current_handler query at line 448)
Lines 448-458 (the handler lookup query) may fail when `current_handler_role` doesn't match any user. No change needed since it already uses `if (handlerUser)` guard.

### Fix (Step 4 — Add null guards to all nested join accesses)
Search the GET /:id handler for ALL places accessing nested join objects (e.g., `complaint.council_member.name`, `complaint.resolution_note`, etc.) and add optional chaining `?.` where missing. Key locations:
- Line 491: `complaint.assigned_council_member_id === userId` → should be `actualAssignedId === userId` (already handled in Step 2)
